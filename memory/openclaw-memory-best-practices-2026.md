## ⚠️ 原生记忆系统的痛点

### 问题 1：记忆碎片化
- 会话记录分散在多个文件
- 缺乏统一的索引层
- 跨会话记忆连贯性差

### 问题 2：检索低效
- 关键词匹配精度低
- 无法理解语义关联
- 长程记忆检索慢

### 问题 3：Token 成本激增
- **ReAct 循环的"记忆税"**：
  - 每次执行都会将上下文、工具输出、执行结果全部写入内存
  - 下一轮循环开始时，这些内容会被再次读取
  - 20 轮对话后，每条新消息都携带沉重的"历史包袱"
  
- **"呼吸成本"**：
  - 即使用户没有主动交互，Heartbeat（心跳）机制也会消耗 Token
  - 长周期任务中，呼吸成本可能超过实际交互成本

### 问题 4：复杂任务完成率低
- 原生 memory-core 模块复杂任务完成率仅 **35.65%**
- 严重制约长周期场景落地价值

### 方案 2：泳道队列（Lane Queue）系统

**解决问题：** 本地多任务并发导致的文件写入冲突

**工作原理：**
- 强制任务串行执行
- 确保记忆写入的准确性
- 避免并发写入导致的数据损坏

**适用场景：**
- 多任务并发环境
- 需要高可靠记忆写入的场景

## 📝 记忆管理最佳实践（2026 推荐）

### 原则 1：分层记忆

| 层级 | 用途 | 更新频率 |
|------|------|----------|
| **短期记忆** | 当前会话上下文 | 实时 |
| **每日笔记** | 当天原始过程记录 | 每日 |
| **长期记忆** | 提炼后的核心上下文 | 定期沉淀 |
| **专项记忆** | 项目/主题专属笔记 | 按需 |

### 原则 2：定期沉淀

**哈基偷的实践：**
- 每天结束时，回顾当日笔记
- 提取值得保留的内容到 MEMORY.md
- 删除/归档过时的临时笔记

**沉淀标准：**
- ✅ 重大事件、关键判断
- ✅ 观点变化、经验教训
- ✅ 需要长期保留的上下文
- ❌ 临时性、过程性的内容

### 原则 3：命名会话

**适用工具：** Claude Code + OpenClaw 协同

**命令示例：**
```bash
# 开始时给会话命名
claude -n auth-refactor

# 或中途命名
/rename feature-payment-v2

# 之后精准恢复
claude --resume feature-payment-v2
```

**好处：**
- 彻底告别"找不到上次对话"
- 项目上下文精准恢复
- 避免记忆碎片化

### 原则 4：硬性规则 vs 软性偏好

**CLAUDE.md（硬性规则）：**
- 代码风格
- 提交流程
- 项目规范

**Auto Memory（软性偏好）：**
- 喜欢的回答格式
- 常用的调试方法
- 个人习惯和偏好

**最佳实践：**
- 把硬性规则写进 CLAUDE.md
- 让 Auto Memory 自然积累软性偏好
